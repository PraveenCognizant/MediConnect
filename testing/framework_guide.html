MediConnect Automation Framework

A full-stack, enterprise-grade Quality Assurance architecture

Framework Blueprint

This automation suite is designed using industry best practices to ensure high maintainability, scalability, and robust reporting. It tests the MediConnect application comprehensively across both UI and Backend API layers.

UI Architecture

Page Object Model (POM)

All web pages are encapsulated in their own Java classes. This separates test logic from UI elements, meaning if the UI changes, we only update the locators in one centralized place, reducing maintenance overhead drastically.

Behavior

Cucumber BDD

Tests are written in plain English using Gherkin syntax (.feature files). This allows product managers and non-technical stakeholders to understand exactly what is being tested without reading Java code.

API Integration

RestAssured API Layer

To ensure full-stack coverage, the framework hits the Spring Boot backend directly. It validates status codes, JSON payloads, and JWT tokens natively, catching backend regressions before they reach the UI.

Data Management

JSON Data Providers

Hardcoded test data is strictly prohibited. We use Jackson databind to parse central JSON files. The custom JsonDataProviders engine natively handles dynamic placeholders like [TIMESTAMP] at runtime.

Environment

Dynamic Configuration

A central config.properties file dictates whether tests run against a local backend or a deployed frontend (e.g. Vercel). The ConfigReader utility manages this seamlessly, allowing zero-code environment swapping.

Code Structure Explained

The framework follows a strict modular directory structure to ensure Separation of Concerns. Here is a deep dive into the folders and their purpose:

src/
└── test/
    ├── java/com/mediconnect/
    │   ├── api/          // RestAssured backend API test classes
    │   ├── base/         // DriverFactory, BaseTest (Core engine logic)
    │   ├── dataprovider/ // JsonDataProviders (TestNG data feeding engine)
    │   ├── listeners/    // TestAuditListener (Allure attachments & logs)
    │   ├── pages/        // Page Object Model classes (Locators & methods)
    │   ├── runner/       // CucumberTestRunner (BDD execution entry point)
    │   ├── steps/        // Cucumber Step Definitions (Glues features to pages)
    │   ├── tests/        // Direct TestNG UI execution classes
    │   └── utils/        // ConfigReader (Environment config management)
    └── resources/
        ├── features/     // Cucumber BDD .feature files in Gherkin
        ├── testdata/     // Centralized JSON test data sets
        └── config.properties // Central environment URLs (UI & API)
/base & /utils

The Core Engine

Contains DriverFactory.java which dynamically spawns and destroys WebDrivers via ThreadLocal, ensuring 100% thread safety for parallel execution. The ConfigReader.java parses the central environment properties.

/pages

The Page Object Layer

Every web page is mapped into a Java class here. The tests interact strictly with these classes, abstracting raw Selenium commands (like driver.findElement) away from the actual test flows.

/steps & /features

The BDD Layer

The .feature files define business behavior. The /steps classes contain the Java code that executes when a Cucumber step (Given, When, Then) is read, acting as the bridge between English instructions and the POM layer.

/api

The Backend Layer

Completely bypasses the browser. These RestAssured classes hit the application's REST endpoints directly to validate server-side logic, database interactions, and authentication natively.

Running the Tests

The framework is highly modular. You can run all tests together via Maven, or isolate execution layers by running specific TestNG classes or Cucumber runners.

Run Everything (CI/CD Mode)

To run both UI and API tests together and generate an Allure report:

mvn clean test

To view the beautiful Allure HTML report afterwards:

mvn allure:serve
Selenium UI Only

Run TestNG UI Tests

Bypass the Cucumber layer and only run the direct Selenium UI tests directly through Surefire:

mvn test -Dtest="com.mediconnect.tests.*"
Cucumber Only

Run BDD Feature Files

To run only the BDD feature files via the Cucumber Runner:

mvn test -Dtest=CucumberTestRunner
RestAssured Only

Run Backend API Tests

If you just made backend changes and want lightning-fast validation without launching a browser:

mvn test -Dtest="com.mediconnect.api.*"
